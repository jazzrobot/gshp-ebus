# Proto v1 Bench Test Record — 2026-03-15

Follow-up: this initial failed bring-up was later succeeded by a clean live capture on
[2026-03-21](2026-03-21-live-capture.md).

## Summary

This was the first full bench bring-up of the `proto-v1` bus-powered breadboard front-end
with the Waveshare `ESP32-S3-ETH` listener.

The session proved:

- the firmware boots and logs correctly
- the logic-side receive path works
- the bridge rectifier works
- the bus-powered `+5V_EBUS` rail works on the live bus

The session did **not** achieve:

- clean live eBUS capture

The main conclusion is that the **schematic remains sound**, but the first breadboard layout
was too pin-centric and too awkward to inspect. The next session should rebuild the
comparator/opto area around clearly separated named nets.

## Setup under test

- board: Waveshare `ESP32-S3-ETH`
- firmware: current passive listener in [../../../firmware/esp32-s3-poe/current/README.md](../../../firmware/esp32-s3-poe/current/README.md)
- listener RX pin: `GPIO16`
- logic power: `3V3` from ESP32 pin `36`, `GND` from ESP32 pin `38`
- front-end type: `proto-v1` bus-powered breadboard design from [../front-end.md](../front-end.md)

## Bring-up sequence and results

### 1. Firmware bring-up

- Initial upload failed because the toolchain had the wrong chip family selected (`ESP32-H2` instead of `ESP32-S3`).
- Upload succeeded after switching Arduino IDE to the correct `ESP32S3 Dev Module` path for the Waveshare board.
- Firmware booted and produced stable periodic `[stats]` output with no input connected.

Observed idle output:

```text
[stats] up_ms=5267 bytes=0 frames=0 trunc=0 frame_err=0 parity_err=0 fifo_ovf=0 buffer_full=0 breaks=0 unknown=0
```

### 2. Logic-side path test

- With no live bus attached, forcing `EBUS_RX` low on the logic side produced `break` events and `0x00` bytes.
- This proved the following chain:
  - `R8` pull-up
  - `R9` series link
  - `ESP_RX`
  - `GPIO16`
  - firmware logging path

Representative output:

```text
[rx] t_us=233533339 byte=0x00 total=1
[uart] event=break count=1
[frame] reason=idle_timeout start_us=233533339 end_us=233533339 len=1 data=00
```

### 3. Live bus power-up

- The live eBUS was attached to the raw bridge input nodes.
- The bus-side supply came up correctly.

Measured values:

- right-hand logic rails: correct `3.3V`
- left-hand rails with no bus attached: `0V`
- `+5V_EBUS`: about `5.25V`
- `+VBUS`: about `19V`
- `EBUS_RX`: about `3.3V` idle

These readings proved:

- the bridge rectifier was wired correctly
- the resistor-zener supply was functioning
- the logic side remained isolated and idle-high

### 4. Layout faults discovered during debug

The first breadboard layout made it too easy to mis-wire the comparator section. During the
session, the following faults were found and corrected or noted:

- `LM393` `pin 8` (`VCC`) had initially been left floating
- `LM393` `pin 5` had initially been tied to `GND_EBUS` instead of `VREF`
- `VREF` and `OUT1` were too easy to confuse in the existing pin-centric placement
- the `VREF` node was too awkward to inspect cleanly with the current layout

One especially important symptom was an invalid `VREF` reading near the rail before the
layout was corrected:

- early `VREF` reading: about `5.2V` instead of the expected `~1.67V`

That reading was used as a sign that the comparator section could not yet be trusted.

### 5. End-of-day behaviour

After partial clean-up of the comparator wiring, the system still did not produce clean live
eBUS traffic.

End-of-day behaviour:

- one boot-time `0x00` / `break` event
- then silence
- no sustained valid byte stream from the live bus

Representative output:

```text
gshp-ebus proto-v1 passive listener
[boot] board=waveshare_esp32_s3_eth rx_pin=GPIO16 usb_baud=115200
[boot] passive mode only; no TX pin configured
[boot] listener armed on UART1 RX=GPIO16 baud=2400 mode=2400-8N1 tx=unassigned
[rx] t_us=1061316 byte=0x00 total=1
[uart] event=break count=1
[frame] reason=idle_timeout start_us=1061316 end_us=1061316 len=1 data=00
```

## Conclusions

- The firmware is in a good enough state for further hardware bring-up.
- The logic-side interface is proven.
- The bus-side power section is proven.
- The `proto-v1` schematic is still the reference design.
- The first breadboard layout is not a good basis for further debugging.

## Next actions

- Rebuild the front-end in a **node-first** layout.
- Give each of these nets its own clear, probeable area:
  - `+VBUS`
  - `GND_EBUS`
  - `+5V_EBUS`
  - `VREF`
  - `SENSE`
  - `U1A_OUT`
  - `EBUS_RX`
  - `ESP_RX`
- Only after the nets are established, jumper them into the `LM393` and `PC817` pins.
- Re-run the same staged checks:
  - logic-side forced-low test
  - `+5V_EBUS`
  - `VREF`
  - `SENSE`
  - live serial capture
